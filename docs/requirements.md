> **文書バージョン**: 1.0
> **最終更新日**: 2026-03-17
> **ステータス**: レビュー中

## 1. システム概要

| 項目 | 内容 |
|------|------|
| システム名 | 見積徴収管理システム（仮称） |
| 目的 | 協力会社への見積依頼・回収・比較を一元管理し、工数80%削減を実現する |
| 発注元 | 三和建設株式会社 |
| 利用者 | 社内26名（東京本店・大阪本店 工務グループ）+ 協力会社 約500社 |
| 月額運用コスト | 約¥3,000〜6,000（Cloud Run + Cloud SQL + Workspace活用） |

### 1.1 解決する課題

| 現状（Before） | 理想（After） |
|---------------|-------------|
| 見積依頼がメール＋Excel添付の手作業 | 管理画面からボタン操作で一括依頼 |
| 進捗管理が個人に依存 | ステータスがリアルタイムで可視化 |
| 毎日1時間以上の工数（ファイル準備・メール・転記） | 工数80%削減 |
| 見積依頼済/受領済のステータス把握が困難 | ダッシュボードで一目瞭然 |
| 比較表の手動作成 | 自動集計＋Excel出力 |
| 転記ミスとメール漏れ | システムによる自動化で撲滅 |

## 2. ユーザーと権限

| ユーザー種別 | 人数 | 認証方式 | 権限 |
|-------------|------|---------|------|
| 社内メンバー | 26名 | Google OAuth（Workspace） | 全機能利用可（権限差なし） |
| 協力会社 | 約500社 | トークン付きURL（ログイン不要） | 自社の見積入力・閲覧・提出のみ |

### 2.1 協力会社のUX原則

- **Googleアカウントの作成を求めない（厳禁）**
- 「送られてきたURLを開く → 入力する → 閉じる」だけで完結
- 事務員によるITサポートが発生しない、極めて直感的なUI
- トークン付きURLにより「推測不可能」＆「期限付き」でセキュリティ確保
- 全世界公開状態は絶対に避ける

## 5. 見積シートのデータ構造（三和書式）

### 5.1 テンプレート構造

テンプレートは工種ごとに用意され、構造は統一されている。
1つのテンプレート内に**複数の棟・エリアシート**が含まれる場合がある（例: 一般倉庫棟、危険物倉庫棟、外構）。

### 5.2 見積明細のカラム定義

| カラム | 説明 | 入力者 | データ型・制約 |
|--------|------|--------|--------------|
| 項目ID | システム自動採番（集計キー） | 自動 | 一意の識別子 |
| 記号 | 項目コード | 三和 | 文字列（空の場合あり） |
| 名称 | 工事項目名 | 三和 | 文字列（必須） |
| 仕様 | 仕様詳細 | 三和 | 文字列 |
| 数量 | 数量 | 三和 | 小数点以下2桁まで（例: 95.50） |
| 単位 | m2, m, 式, ヶ所 等 | 三和 | 文字列 |
| **単価** | **見積単価** | **協力会社** | **整数のみ（小数不可）** |
| **金額** | **自動計算** | **自動** | **floor(数量 × 単価) — 小数点以下切り捨て** |
| 備考 | 補足説明 | 協力会社（任意） | 文字列 |

> **注意**: `記号`は空の場合があるため、集計キーには使用しない。
> システムが自動採番する`項目ID`を内部キーとする。
> 協力会社が行を追加した場合（項目不足・諸経費追加・仕様変更説明）、
> 追加行にも自動で項目IDが付与され、比較表では別行として表示される。

### 5.3 付帯項目（明細末尾）

| 項目 | 説明 |
|------|------|
| 法定福利費 | 料率 × 小計で算出。協力会社が率または金額を入力 |
| 養生費・運搬諸経費 | 協力会社が金額を入力 |
| 調整費 | 任意。端数調整等 |

### 5.4 見積情報（ヘッダー）

| フィールド | 入力者 |
|-----------|--------|
| 見積番号 | 協力会社 |
| 日付 | 協力会社 |
| 会社名 | 自動（マスタから） |
| 担当者名 | 協力会社 |
| 工事名称 | 自動（案件情報から） |
| 見積金額 | 自動計算 |
| NET金額 | 協力会社 |
| 特記事項 | 協力会社（任意） |

### 5.5 大規模テンプレートへの対応

電気設備テンプレートで約1,562行・1,091項目の実績があり、
UIは仮想スクロールまたはセクション単位のページネーションで対応する。

## 6. 比較表

### 6.1 構造

| 列グループ | 内容 |
|-----------|------|
| 基本情報 | 記号・名称・仕様・数量・単位 |
| 予算 | 予算単価・予算金額 |
| 各社（N社分） | 数量・単位・単価・金額・備考・該当予算 |

### 6.2 仕様

- **N社横並び比較**: 3社制限なし（実績: 4社+予算列の並列表示）
- **棟・エリアごとにグループ化**
- **末尾に法定福利費・諸経費・調整費を社別に表示**
- **会社選択**: チェックボックスで比較対象の会社を選択
  - パフォーマンスに問題がなければリアルタイム比較
  - 重い場合は「比較する」ボタンクリックで比較実行
- **Excel出力**: 必須（現在の比較表フォーマット準拠）
- **画面上での比較表示**: 必須

## 7. ステータス管理

```
未発行 ──→ 依頼中 ──→ 下書き中 ──→ 提出済 ──┬──→ 確定（発注）
                      （協力会社が       │
                        入力中）         ├──→ 不採用
                                         │
                                         └──→ 差し戻し ──→ 下書き中（再入力）
```

| ステータス | 説明 | 操作者 |
|-----------|------|--------|
| 未発行 | テンプレート準備中 | 社内 |
| 依頼中 | 見積依頼メール送信済み | 自動 |
| 下書き中 | 協力会社が入力を開始（自動検知） | 自動 |
| 提出済 | 協力会社が提出ボタンを押下 | 協力会社 |
| 確定（発注） | 採用決定 | 社内 |
| 不採用 | 不採用決定 | 社内 |
| 差し戻し | 修正依頼 | 社内 |

## 8. 差し戻しフロー

| 拠点 | 差し戻し方法 |
|------|-------------|
| 東京 | 修正箇所と内容をコメントで添付 |
| 大阪（大幅変更） | 変更図面＋変更リストをファイル添付 |
| 大阪（部分変更） | 変更内容テキスト＋変更図面をファイル添付 |

→ システムでは **コメント入力 + ファイル添付（複数可）** の両方に対応

## 12. フェーズ計画

### Phase 1（MVP）— 三和書式の見積フロー

- 案件管理（CRUD）
- 協力会社マスタ管理（CRUD + CSVインポート）
- 見積テンプレート管理（Excelインポート → システムにデータ化）
- 見積依頼の一括発行（メール送信）
- 協力会社ポータル（トークン付きURL、単価入力フォーム）
- ステータス管理（全6状態 + 差し戻し）
- 比較表（自動集計 + 会社選択 + 画面表示 + Excel出力）
- 通知（依頼・提出完了・リマインド・差し戻し）

### Phase 2 — 協力会社書式への対応

- 協力会社書式の見積依頼フロー
- ファイルアップロード機能（Excel/PDF）
- Google Driveへのファイル保存
- アップロード済ファイルの閲覧・ダウンロード

### Phase 3 — 高度機能

- OCR（Google Document AI）による協力会社書式のデータ化
- OCRデータ → 手動マッピング → 比較表への取り込み
- リマインド自動化の高度化（段階的リマインド等）
- iNDEX連携の自動化（必要に応じて）

> ※ フェーズの優先順位は開発を進めながら変更の可能性あり

## 13. 非機能要件

| 項目 | 要件 |
|------|------|
| レスポンス | 画面遷移 2秒以内、比較表生成 5秒以内 |
| 同時利用 | 最大30名程度（社内26名 + 協力会社数名の同時アクセス） |
| データ保存 | 過去の見積データに保存期限の制約なし（無期限保存） |
| セキュリティ | トークンは推測不可能な暗号学的ランダム値、有効期限付き |
| 可用性 | 業務時間帯（平日9:00-18:00）に安定稼働 |
| バックアップ | Cloud SQLの自動バックアップ（日次） |
